# <a name="loading-server-capacity-planning-worksheet-for-analytics-platform-system"></a>Загрузка таблица планирования емкости сервера для Analytics Platform System
Это таблица планирования емкости поможет вам определить требования для сервера для загрузки для загрузки данных в SQL Server PDW. Используется для создания плана для приобретения или подготовки существующих загрузка серверов.  
  
## <a name="worksheet-notes"></a>Заметки о листе
  
1.  На этом листе относится к серверам, будет загрузка данных с помощью **dwloader** программа командной строки для загрузки.  
  
2.  Для загрузки данных с помощью служб Integration Services или сторонним инструмента загрузка сторона, требования различаются в зависимости от различий в процессе загрузки.  
  
3.  Применить большинство требований к загрузке файлов либо сжатом или несжатом данных. любые различия в требованиях указаны полужирным шрифтом.  
  
## <a name="clipboardmediaclipboard-iconpng-clipboard-capacity-planning-worksheet"></a>![Буфер обмена](media/clipboard-icon.png "буфер обмена") планирование мощности листа  
  
Печать на этом листе и наполнить ваших собственных требований.  
  
|Компонент|Требование|Заполните этот столбец с требованиями|Рекомендации|  
|-------------|---------------|--------------------------------------------------|-------------------|  
|Память|Максимальное число байтов, которые планируется хранить на сервере загрузки в заданный период времени.|![Значок с изображением карандаша](media/pencil-icon.png "значок с изображением карандаша")|Чтобы определить требования к хранилищу, выясните, какой объем данных планируется хранить на сервере загрузки в заданный период времени.  Требования к емкости, нагрузки только для файлов; операционная система и загружать файлы должны располагаться на разных дисковые массивы.<br /><br />Пример: Если вы планируете загрузить 100 ГБ данных с диска в 3 раза каждый день, но не удаляет файлы данных, до конца недели, то вы должны минимальное 2.1 ТБ для хранения файлов данных. Мы рекомендуем благосостояние осторожен и получении о 30% больше хранилища для учетной записи для отклонения и роста.  Например лучше 2.73 ТБ дискового пространства.|  
|Скорость загрузки|Максимальное число байтов в час для данных для загрузки в PDW.|![Значок с изображением карандаша](media/pencil-icon.png "значок с изображением карандаша")|Это является примерным. При вычислении это требование, предполагается, что файлы уже находятся на сервере загрузки и, как можно лучше других нагрузке.<br /><br />Пример: Не нужно учитывать в сжимаемости данных, так как dwloader всегда отправляет несжатых данных в PDW. Не нужно учитывать в преобразованиях типов данных и размер целевой таблицы.|  
|Сеть|Тип сетевого соединения.|![Значок с изображением карандаша](media/pencil-icon.png "значок с изображением карандаша")|Определите наилучший тип сетевого подключения в соответствии с требованиями скорость загрузки.<br /><br />Пример: InfiniBand или 10 Гбит Ethernet будет предоставить сведения о тарифах оптимальной загрузки. Ethernet 1 Гбит ограничит нагрузки ставкам до 360 ГБ в час или менее.|  
|Ввод-вывод|Байты в час для операций чтения и записи.|![Значок с изображением карандаша](media/pencil-icon.png "значок с изображением карандаша")|Чтобы загрузить данные, dwloader должны считать все данные из диска перед отправкой в PDW.<br /><br />Каждый сервер загрузки не могут загружать данные быстрее, чем устройства могут получать данные из всех источников для загрузки. Чтобы сэкономить деньги, планирование емкости для загрузки, чтобы его значение не превышает емкость устройства чтения ввода-вывода.<br /><br />Пример:<br />PDW получает и загружает данные в это устройство 1 стойка с максимальной скоростью 1,8 ТБ в час. Для устройства с 2 или более стойками коэффициент максимальной нагрузки — 3,6 ТБ в час.<br /><br />Если вы планируете загрузить из нескольких серверов загрузки, в то же время, требования ввода-вывода для каждого сервера загрузки будет меньше, чем при одном сервере все загрузки.<br /><br />Пример: Один сервер загрузки можно загрузить более 1,8 ТБ в час для модуля, стойка 1. Двух серверов загрузки каждого одновременно загрузить 900 ГБ в час в устройство стойки 1. Более высокий уровень параллелизма может снизить эффективность и максимальную пропускную способность.<br /><br />Производительность ввода-вывода принимая в учетной записи всех операций ввода-вывода на сервере загрузки. Если загрузка сервера имеет другой трафик ввода-вывода, в дополнение к загрузке данных, таких как прием данных файлов с сервера ETL, увеличит требования ввода/вывода.<br /><br />Для сжатых данных ввода-вывода требования зависят от степень сжатия данных. dwloader будет считывать сжатые данные и затем распаковывает его перед отправкой в PDW. Чем выше степень сжатия, тем меньше данных загрузка сервера потребуется, считанных с диска.<br /><br />Пример: Если скорость необходимые нагрузки является 1,8 ТБ в час, а данные хранятся на сервере загрузки с использованием сжатия 2:1, загрузка сервера требуется только для чтения 900 ГБ в час с диска, а не 1,8 ТБ. Коэффициент сжатия 3:1 означает, что загрузка сервера должен считать 600 ГБ в час с диска.|  
|ЦП|Число процессоров.|![Значок с изображением карандаша](media/pencil-icon.png "значок с изображением карандаша")|Для загрузки несжатых данных, dwloader не является приложением интенсивно использующие ЦП.  Как минимум рекомендуется использовать двухпроцессорный сервер недавно производства.<br /><br />Загрузка сжатых данных, вам потребуется достаточное количество ЦП, чтобы распаковать данные перед отправкой в PDW. dwloader можно одновременно запустить 10 активных потоков. Если планируется загружать 10 сжатых файлов параллельно, мы рекомендуем, что сервер имеет по крайней мере 10 ядрами ЦП или двумя 6-ядерными процессорами.|  
|ОЗУ|Загружает ГБ памяти, который позволяет кэшировать файлы во время Windows.|![Значок с изображением карандаша](media/pencil-icon.png "значок с изображением карандаша")|dwloader использует очень мало ОЗУ на сервере загрузки. Для производительности Windows использует память для кэширования файлов загрузки после их чтения с диска.<br /><br />Определить требования к оперативной памяти, см. в установке Windows Server и требования приложения сторонних производителей. Рекомендуется не менее 32 ГБ, если у вас нет требования из других источников.<br /><br />Для сжатых данных быстрее ОЗУ полезно, так как он позволяет ускорить распаковки.|
