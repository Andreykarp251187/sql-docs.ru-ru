# <a name="loading-server-capacity-planning-worksheet-for-analytics-platform-system"></a>Загрузка листа планирования емкости сервера для системы аналитики платформы
Этот лист планирования емкости помогает определить требования к серверу загрузки для загрузки данных в SQL Server PDW. Используйте этот параметр, чтобы создать план для приобретения или подготовки существующих серверов загрузки.

## <a name="worksheet-notes"></a>Заметки листа

1.  Этот лист применяется к серверам, которые будут загружать данные с помощью средства загрузки из командной строки **dwloader** .

2.  Для загрузки данных с помощью Integration Services или средства загрузки сторонних производителей требования могут различаться в зависимости от различий в процессе загрузки.

3.  Большинство требований применимо для загрузки сжатых или несжатых файлов данных. все различия в требованиях отмечены полужирным шрифтом.

## <a name="clipboard-capacity-planning-worksheet"></a>![Буфер обмена](media/clipboard-icon.png "Буфер обмена") Лист планирования ресурсов

Распечатайте этот лист и заполните его собственными требованиями.

|Компонент|Требование|Заполните этот столбец собственными требованиями|Рекомендации|
|-------------|---------------|--------------------------------------------------|-------------------|
|Память|Максимальное число байт, которое планируется хранить на сервере загрузки в любой конкретный период времени.|![Значок "Карандаш"](media/pencil-icon.png "Значок "Карандаш"")|Чтобы определить требования к хранилищу, определите объем данных, которые планируется хранить на сервере загрузки в любой конкретный период времени.  Требования к емкости предназначены только для загрузки файлов. операционная система и файлы загрузки должны находиться на разных дисковых массивах.<br /><br />Например, если планируется загружать 100 ГБ данных с диска 3 раз в день, но не удалять файлы данных до конца недели, то для хранения файлов данных потребуется минимум 2,1 ТБ. Мы рекомендуем использовать консервативные и более 30% дополнительных объемов памяти, чтобы учитывать дисперсию и рост.  В этом примере лучше использовать 2,73 ТБ дискового пространства.|
|Скорость загрузки|Максимальное число байтов в час для загрузки данных в PDW.|![Значок "Карандаш"](media/pencil-icon.png "Значок "Карандаш"")|Это приблизительная оценка. При вычислении этого требования следует предположить, что файлы уже находятся на сервере загрузки, а другие условия загрузки максимально приемлемы.<br /><br />Например: не нужно учитывать сжимаемости данных, так как dwloader всегда отправляет в PDW несжатые данные. Нет необходимости разделять преобразования типов данных и размер целевой таблицы.|
|Сеть|Тип сетевого подключения.|![Значок "Карандаш"](media/pencil-icon.png "Значок "Карандаш"")|Определите оптимальный тип сетевого подключения для требований к скорости загрузки.<br /><br />Например: InfiniBand или 10 Гбит/с Ethernet обеспечит оптимальную скорость загрузки. 1 Гбит/с Ethernet ограничит скорости загрузки до 360 ГБ в час или меньше.|
|Ввод-вывод|Байт в час для операций чтения и записи.|![Значок "Карандаш"](media/pencil-icon.png "Значок "Карандаш"")|Чтобы загрузить данные, dwloader должен прочитать все данные с диска перед их отправкой в PDW.<br /><br />Каждый сервер загрузки не может загружать данные быстрее, чем устройство может принимать данные из всех источников загрузки. Чтобы сэкономить деньги, спланируйте емкость операций ввода-вывода для загрузки, чтобы она не превышала мощность нагрузки устройства.<br /><br />Пример:<br />PDW получает и загружает данные в устройство с 1 стойкой с максимальной скоростью 1,8 ТБ в час. Для устройства с двумя и более стойками максимальная скорость загрузки составляет 3,6 ТБ в час.<br /><br />Если планируется загрузка с нескольких серверов загрузки одновременно, требования к вводу-выводу для каждого сервера загрузки будут меньше, чем когда один сервер выполняет всю загрузку.<br /><br />Например: один сервер загрузки может загружать не более 1,8 ТБ в час для одного устройства в стойку. Два сервера загрузки могут параллельно загружать 900 ГБ в час в одно устройство с 1 стойками. Более высокие уровни параллелизма могут снизить эффективность и максимальную пропускную способность.<br /><br />Для емкости операций ввода-вывода необходимо учитывать все операции ввода-вывода, происходящие на сервере загрузки. Если сервер загрузки имеет другой трафик ввода-вывода в дополнение к загрузке данных (например, получение файлов данных с ETL-сервера), требования ввода/вывода будут увеличены.<br /><br />Для сжатых данных требования к вводу-выводу зависят от скорости сжатия данных. dwloader считывает сжатые данные, а затем распаковывает их перед отправкой в PDW. Чем выше коэффициент сжатия, тем меньше данных будет необходимо считывать сервер загрузки с диска.<br /><br />Например, если требуемая скорость загрузки составляет 1,8 ТБ в час, а данные хранятся на сервере загрузки с 2:1-сжатием, то сервер загрузки должен только считывать с диска только 900 ГБ в час, а не 1,8 ТБ. Коэффициент сжатия 3:1 означает, что сервер загрузки должен считывать 600 ГБ в час с диска.|
|ЦП|Число сокетов.|![Значок "Карандаш"](media/pencil-icon.png "Значок "Карандаш"")|Для загрузки несжатых данных dwloader не является приложением с интенсивным использованием ЦП.  В качестве минимального требования рекомендуется использовать недавно изготовленный сервер с двумя сокетами.<br /><br />Для загрузки сжатых данных необходима достаточная мощность ЦП для распаковки данных перед их отправкой в PDW. dwloader может запускать 10 активных потоков одновременно. Если планируется параллельная загрузка 10 сжатых файлов, рекомендуется, чтобы на сервере было по крайней мере 10-ядерный ЦП или два 6-ядерных ЦП.|
|ОЗУ|ГБ памяти, которая позволяет Windows кэшировать файлы во время загрузки.|![Значок "Карандаш"](media/pencil-icon.png "Значок "Карандаш"")|dwloader использует очень мало ОЗУ на сервере загрузки. Для повышения производительности Windows использует память для кэширования загружаемых файлов после их чтения с диска.<br /><br />Чтобы определить требования к ОЗУ, обратитесь к установке Windows Server и всем требованиям сторонних приложений. Если у вас нет требований из других источников, рекомендуется не менее 32 ГБ.<br /><br />Для сжатых данных удобнее использовать ОЗУ, так как это ускоряет распаковку.|
