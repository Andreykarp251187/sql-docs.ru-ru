# <a name="backup-server-capacity-planning-worksheet---parallel-data-warehouse"></a>Лист планирования емкости резервного сервера — Параллельное хранилище данных
Этот лист планирования емкости помогает определить требования к резервному серверу для выполнения операций резервного копирования и восстановления базы данных SQL Server PDW. Используйте этот параметр, чтобы создать план для приобретения новых или подготовки существующих резервных серверов.

Этот лист является дополнением к инструкциям в статье [Получение и Настройка резервного сервера](acquire-and-configure-backup-server.md).

## <a name="capacity-planning-worksheet-for-backup-servers"></a>Лист планирования ресурсов для резервных серверов

### <a name="notes"></a>Примечания

1.  Этот лист применяется к серверам, которые будут выполнять операции резервного копирования и восстановления для баз данных PDW.

2.  Единственным способом резервного копирования и восстановления баз данных PDW является использование команд BACKUP DATABASE и RESTORE DATABASE SQL. Однако после того, как данные резервной копии находятся на резервном сервере, они существуют как набор файлов Windows. Вы можете архивировать файлы резервных копий с сервера в другое место хранения с помощью традиционных методов резервного копирования на основе файлов Windows.

### <a name="clipboard-icon-capacity-planning-worksheet"></a>![Значок буфера обмена](media/clipboard-icon.png "Значок буфера обмена") Лист планирования ресурсов 

Распечатайте этот лист и заполните его собственными требованиями.

|Компонент|Требование|Заполните этот столбец собственными требованиями|Рекомендации|
|-------------|---------------|--------------------------------------------------|-------------------|
|Память|Максимальное количество байт, которое планируется хранить на сервере архивации в любой конкретный период времени.|![Значок "Карандаш"](media/pencil-icon.png "Значок "Карандаш"")|Чтобы определить требования к хранилищу, определите объем данных, которые планируется хранить на сервере архивации в любой конкретный период времени.<br /><br />Резервные данные хранятся на сервере архивации в сжатом формате. Скорость сжатия данных зависит от характеристик данных.<br /><br />Например, в качестве приблизительной оценки мы рекомендуем оценивать коэффициент сжатия 7:1 относительно размера несжатых данных. Это предполагает, что по крайней мере 80% данных хранятся в кластеризованных индексах columnstore. Например, если в базе данных имеется 700 ГБ несжатых данных и они хранятся в кластеризованных индексах columnstore, то для резервной копии базы данных потребуется около 100 ГБ.<br /><br />Если на резервном сервере планируется создать несколько копий резервных копий баз данных, их необходимо учитывать.<br /><br />Например, если вы планируете создать резервную копию 10 баз данных, каждая из которых содержит 5 ТБ несжатых данных, размер баз данных будет 50 ТБ. Если вы планируете создавать резервные копии этих 10 баз данных ежедневно в течение 5 дней в строке, общий размер без сжатия составляет 250 ТБ. При анализе коэффициента сжатия 7:1 вам потребуется 250/7 = 35,7 ТБ хранилища на резервном сервере. Рекомендуется консервативнь и получить около 30% дополнительных мощностей, чтобы учитывать дисперсию и рост.  В этом примере 46,6 ТБ будет лучше.|
|Сеть|Тип сетевого подключения.|![Значок "Карандаш"](media/pencil-icon.png "Значок "Карандаш"")|Определите оптимальный тип сетевого подключения, который может соответствовать требованиям к скорости загрузки.<br /><br />Например: InfiniBand или 10Gbit Ethernet обеспечит оптимальную скорость загрузки. Рамках 1 Ethernet ограничит скорость загрузки до 360 ГБ в час или меньше.|
|Ввод-вывод|Байт в час для операций записи.|![Значок "Карандаш"](media/pencil-icon.png "Значок "Карандаш"")|Для записи резервных копий на диск скорость записи по 4 ТБ в час является оптимальной.<br /><br />Например, для дисков, которые могут записывать 50 МБ/с, вам потребуется по крайней мере 24 диска, а также дополнительные сведения для зеркального отображения или четности.<br /><br />Для емкости операций ввода-вывода необходимо учитывать все операции ввода-вывода, происходящие на сервере загрузки. Если сервер загрузки имеет другой трафик ввода-вывода в дополнение к загрузке данных (например, получение файлов данных с ETL-сервера), требования ввода/вывода будут увеличены.|
|ЦП|Число сокетов.|![Значок "Карандаш"](media/pencil-icon.png "Значок "Карандаш"")|Получение и хранение файлов резервных копий не является приложением с интенсивным использованием ЦП.  В качестве минимального требования рекомендуется использовать недавно изготовленный сервер с двумя сокетами.|
|ОЗУ|ГБ памяти, которая позволяет Windows кэшировать файлы во время загрузки.|![Значок "Карандаш"](media/pencil-icon.png "Значок "Карандаш"")|Для получения и хранения файлов резервных копий требуется очень мало ОЗУ на сервере загрузки.<br /><br />Чтобы определить требования к ОЗУ, обратитесь к установке Windows Server и всем требованиям сторонних приложений. Если у вас нет требований из других источников, рекомендуется не менее 32 ГБ.|

По завершении определения требований к емкости вернитесь к разделу [Получение и настройка сервера загрузки](acquire-and-configure-loading-server.md) , чтобы спланировать покупку.
