# <a name="analysis-services-developer-documentation"></a>Документация для разработчика служб Analysis Services
[!INCLUDE[ssas-appliesto-sqlas-aas](../includes/ssas-appliesto-sqlas-aas.md)]

В службах Analysis Services почти каждый объект и рабочей нагрузки можно программировать и часто есть более чем один подход на выбор.  Возможные варианты написания управляемого кода, сценария или если ваши требования к решению исключают, с помощью .NET framework, с использованием открытых стандартов, например XMLA и MSOLAP.

## <a name="what-you-can-accomplish-in-code"></a>Что можно сделать в коде
Типичные сценарии программирования включают сервера и развертывания базы данных, администрирования, модели и создания базы данных и доступ к данным из пользовательских приложениях и отчеты, которые используют данные служб Analysis Services. Общие для всех этих сценариях предопределенной архитектуры и объект определения иерархии, понятных операции, охватывающие определение данных, обработки и рабочих нагрузок запросов.

Несмотря на то, что объекты и рабочие нагрузки являются программируемыми, они не являются расширяемыми. В частности нельзя создать картриджей пользовательские данные, которые получают данные из источников данных не поддерживается, настройки или замена поведение ядра формулу или хранения, а также можно создавать новые типы объектов метаданных на сервере, базе данных или модели.

Дополнительно уточнять последней точки о создании новых типов объектов: нельзя создавать новый тип объекта, можно создать вычисляемые объекты, полученные из выражениях или коде во время выполнения. Не все, что в модели должен быть заранее и сопоставляется с существующей структуры данных. Кроме того можно расширить схему с помощью заметок в объектах AMO для передачи сведений об объекте в клиентском приложении.

## <a name="choose-a-platform-or-approach-to-development"></a>Выбор платформы или подход к разработке
Службы Analysis Services предоставляют много способов настройки решения с помощью кода, но большинство разработчиков используют управляемые API-интерфейсы или скрипт.

- Управляемый API включают в себя [TOM и AMO](http://msdn.microsoft.com/library/mt436122.aspx) для определения данных и административные задачи, и [ADOMD.NET](http://msdn.microsoft.com/library/mt465769.aspx) для поддержки запросов из клиентского кода. В SQL Server 2016 AMO обновляется для использования новых табличных метаданных для моделей создана или обновлена до уровня совместимости 1200 и выше.

- Скрипт, часто можно добиться те же результаты, что исполняемый файл с возможно меньше работы программы.

  - Можно написать скрипт PowerShell с помощью компонентов служб Analysis Services PowerShell, которые напрямую вызывают типов объектов AMO. В PowerShell можно также создать и выполнить ASSL/XMLA или сценарий TMSL (в формате JSON).

  - ASSL и TMSL являются языков скриптов, которые предоставляют объекты, используемые в поиска и выполнения операций. Использования какого рода сценарий зависит от базового сервера, базы данных или модели.

  - Табличные модели или базы данных на уровне совместимости 1200 и выше использовать модель языка скриптов табличной (TMSL), который находится в JSON.

  - Использовать Analysis Services Scripting Language (ASSL), которой — это расширение в открытом стандарте XMLA служб Analysis Services, многомерных и табличных моделей на уровни совместимости 1050 – 1103.

  - В среде Management Studio можно создать скрипт ASSL или TMSL. Можно также использовать **просмотреть код** в SQL Server Data Tools для просмотра определения модели в ASSL и TMSL.

- Хотя это можно выполнить сборку решения, основанные на стандартах open XML для Аналитики и многомерных Выражений, это крайне редко, чтобы сделать это. Документация отсутствует, отличные от XML для Аналитики и справочник по многомерных Выражений, чтобы помочь вам и большинство сообщества и форум поддержки выводит из работы с помощью .NET или собственными технологиями (MSOLAP).

## <a name="programming-in-analysis-services"></a>Программирование в службах Analysis Services
[Программирование интеллектуального анализа данных](../analysis-services/data-mining-programming.md) описывает методы построения решений, которые включают объекты интеллектуального анализа данных.

[Программирование многомерной модели](../analysis-services/multidimensional-models/multidimensional-model-programming.md) описывает задачи разработки и способы интеграции объектов многомерной модели в пользовательском решении.

[Программирование табличных моделей с уровнем совместимости 1200 и выше](../analysis-services/tabular-model-programming-compatibility-level-1200/tabular-model-programming-for-compatibility-level-1200.md)
**нового в SQL Server 2016**.  Обзор интерфейсов и языков скриптов, используемых для работы с Tabular 1200 и выше моделей программным способом.

[Программирование табличных моделей для совместимости уровни 1050 по 1103](../analysis-services/tabular-model-programming-compatibility-levels-1050-1103/tabular-model-programming-for-compatibility-levels-1050-through-1103.md) эта документация предназначена для разработчиков, которые поддерживают табличные модели на более низких уровнях совместимости. Он описывает расширения языка CSDL, определяющие табличной модели в синтаксисе XML. Он также включает сведения о синтаксисе и определения модели табличного объекта.

[Объекты управления Analysis Services (AMO)](https://msdn.microsoft.com/library/mt436122.aspx) документация по справочнику разработчика для управляемого поставщика Analysis Services Management объектов (AMO), для определения данных и администрирования, включая обработку.

[ADOMD.NET](http://msdn.microsoft.com/library/mt465769.aspx) документация по справочнику разработчика для управляемого поставщика, ADOMD.NET, используется для программного доступа и отправки запросов рабочих нагрузок.

[Наборы строк схемы служб Analysis](https://docs.microsoft.com/bi-reference/schema-rowsets/analysis-services-schema-rowsets) описывает наборы строк схемы, которые предоставляют сведения о состоянии сервера, операциях сервера и объекты базы данных.

[XML для аналитики &#40;XMLA&#41; ссылку](https://docs.microsoft.com/bi-reference/xmla/xml-for-analysis-xmla-reference) XMLA описаны основные понятия, которые помогут вам понять, как XML для Аналитики помогает реализовать пользовательское решение. Также описывается уровень совместимости со спецификацией XMLA 1.1.

[Язык сценариев служб Analysis Services &#40;ASSL для XMLA&#41; ](https://docs.microsoft.com/bi-reference/assl/analysis-services-scripting-language-assl-for-xmla) описывает расширения языка ASSL для XMLA. Язык ASSL — это язык для определения и работы с данными для многомерных моделей служб Analysis Services, дополняющий спецификацию XMLA.

[Tabular Model Scripting Language &#40;TMSL&#41; ссылку](https://docs.microsoft.com/bi-reference/tmsl/tabular-model-scripting-language-tmsl-reference) TMSL является JSON-представление табличных моделей на уровне совместимости 1200 и выше. Определения объектов строятся на табличных метаданных конструкций, таких как таблицы, столбца и связи, а не многомерные метаданные, которые могут быть не знакомы, если вы не знакомы с моделирования данных служб Analysis Services в табличном режиме.

[Analysis Services PowerShell Reference](../analysis-services/powershell/analysis-services-powershell-reference.md) Документирует командлеты, используемые для административных функций, а также общего назначения **Invoke-ASCmd** командлетом, принимающим любой скрипт или запрос в качестве входных данных.
