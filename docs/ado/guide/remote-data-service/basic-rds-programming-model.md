# <a name="basic-rds-programming-model"></a>Основная модель программирования RDS
> [!IMPORTANT]
>  Начиная с Windows 8 и Windows Server 2012, серверные компоненты служб удаленных рабочих СТОЛОВ, больше не включаются в операционной системе Windows (см. в разделе Windows 8 и [настольная книга по совместимости Windows Server 2012](https://www.microsoft.com/download/details.aspx?id=27416) для получения дополнительных сведений). Клиентские компоненты служб удаленных рабочих СТОЛОВ будет поддерживаться в будущих версиях Windows. Избегайте использования этого компонента в новых разработках и запланируйте изменение существующих приложений, в которых он применяется. Приложения, использующие служб удаленных рабочих СТОЛОВ, следует перевести [WCF-сервиса данных](https://go.microsoft.com/fwlink/?LinkId=199565).  
  
 Служба удаленных рабочих СТОЛОВ Active Directory решают приложений, которые существуют в следующем окружении. Клиентское приложение указывает программу, которая будет выполняться на сервере, а также параметры, необходимые для возвращать нужные сведения. Программа, вызванная на повышение доступа к серверу, для указанного источника данных, извлекает сведения, при необходимости обрабатывает данные и затем возвращает полученные результаты в форме, которую можно легко использовать клиентское приложение. RDS предоставляет средства можно выполнить следующую последовательность действий:  
  
-   Укажите программу для вызова на сервере и получить возможность ссылаться на него из клиента. (Эта ссылка иногда называют *прокси-сервера*. Он представляет программу удаленного сервера. Клиентское приложение будет «call» прокси-сервера, как если бы это был локальный программы, но он фактически вызывает программу удаленного сервера.)  
  
-   Вызовите программу сервера. Передайте параметры для определения источника данных и выдачи команд программы-сервера. (Программа server фактически использует ADO для доступа к источнику данных. ADO устанавливает соединение с одной из заданных параметров, а затем выдает команды, указанной в другой параметр.)  
  
-   Программа server получает [записей](../../../ado/reference/ado-api/recordset-object-ado.md) из источника данных. При необходимости **записей** объект обрабатывается на сервере.  
  
-   Программа server Возвращает конечное **записей** объект клиентскому приложению.  
  
-   На стороне клиента **записей** объект помещается в форму, можно легко использовать визуальные элементы управления.  
  
-   Любые изменения **записей** объект отправляются обратно в программу сервера, который использует их для обновления источника данных.  
  
 Эта модель программирования содержит функции, определенные удобства. Если нет необходимости программу сложного сервера для доступа к источнику данных, и если указать необходимые параметры подключения и команды, служб удаленных рабочих СТОЛОВ будут автоматически найти нужные данные с помощью простой, программы server по умолчанию.  
  
 Если вам требуется более сложную обработку, можно указать собственную программу пользовательского сервера. К примеру так как программа на пользовательский сервер имеет всю мощь ADO в ее распоряжении, его может подключиться к несколько различным источникам данных, объединить свои данные в сложных каким-либо образом и затем возвращают результат простой, обработанные в клиентское приложение.  
  
 Наконец Если вашим потребностям, где-то между ними ADO теперь поддерживает настройки поведения серверной программы по умолчанию.
