# <a name="securing-rds-applications"></a>Защита приложений RDS
Этот раздел содержит сведения о безопасности для RDS.  
  
> [!IMPORTANT]
>  Начиная с Windows 8 и Windows Server 2012, серверные компоненты служб удаленных рабочих СТОЛОВ, больше не включаются в операционной системе Windows (см. в разделе Windows 8 и [настольная книга по совместимости Windows Server 2012](https://www.microsoft.com/download/details.aspx?id=27416) для получения дополнительных сведений). Клиентские компоненты служб удаленных рабочих СТОЛОВ будет поддерживаться в будущих версиях Windows. Избегайте использования этого компонента в новых разработках и запланируйте изменение существующих приложений, в которых он применяется. Приложения, использующие служб удаленных рабочих СТОЛОВ, следует перевести [WCF-сервиса данных](https://go.microsoft.com/fwlink/?LinkId=199565).  
  
## <a name="microsoft-internet-explorer-security-issues"></a>Проблемы безопасности Microsoft Internet Explorer  
 Благодаря новым улучшениям безопасности, добавляемый в Microsoft Internet Explorer некоторые объекты ADO и RDS ограничены только в среде «безопасный режим». Для этого известно об этих проблемах, включая разным зонам, уровни безопасности, строгие поведение, небезопасных операций и пользовательские параметры безопасности.  
  
## <a name="security-and-your-web-server"></a>Безопасность и веб-сервера  
 Если вы используете [RDSServer.DataFactory](../../../ado/reference/rds-api/datafactory-object-rdsserver.md) объекта Internet веб-сервера, помните, что это лишь создает потенциальную угрозу безопасности. Внешние пользователи, которые получить допустимый источник данных (DSN), идентификатор пользователя и сведения о пароле написать страницы для отправки любого запроса к источнику данных. Если требуется более ограниченный доступ к источнику данных, один из вариантов — отменить регистрацию и удалить **RDSServer.DataFactory** (msadcf.dll) и вместо этого использовать настраиваемые бизнес-объекты с жестко запросами.  
  
 Дополнительные сведения об аспектах безопасности с помощью объекта RDSServer.DataFactory см. в разделе по безопасности Microsoft MS99-025 на безопасности веб-узла.  
  
## <a name="client-impersonation-and-security"></a>Олицетворение клиента и безопасность  
 Если **проверку пароля** имеет значение свойства вашего веб-сервера IIS для проверки подлинности Windows NT вызов-ответ (для Windows NT 4.0) или для интеграции Windows проверка подлинности (для Windows 2000), то не бизнес-объектов вызывается в контексте безопасности клиента. Это новая функция в версии 1.5 служб удаленных рабочих СТОЛОВ, позволяющий олицетворение клиента по протоколу HTTP. При работе в этом режиме, вход в систему на веб-сервер (IIS) не является анонимным, но использует идентификатор пользователя и пароль, под которой работает клиентский компьютер. Если ODBC DSN настроены использовать доверительное соединение, затем доступ к базам данных, например SQL Server, также происходит в контексте безопасности клиента. Но это работает, только если база данных находится на том же компьютере, что и службы IIS; учетные данные клиента не переносятся на еще один компьютер.  
  
 Например, клиент, John Doe, userid = «JohnD» и пароль = «секрет» вошел на клиентский компьютер. Он запускает веб-приложение, которому требуется доступ к **RDSServer.DataFactory** создаваемого объекта [записей](../../../ado/reference/ado-api/recordset-object-ado.md) , выполнив запрос SQL на сервере «MyServer» под управлением IIS. MyServer, системы под управлением Windows NT Server 4.0, предназначена для использования проверки подлинности Windows NT вызов-ответ, его имя DSN ODBC содержит «Использовать доверительное соединение» выбран и этот сервер также содержит источник данных SQL Server. При получении запроса на веб-сервере, запрашивает у клиента для пользователя и пароль. Таким образом, запрос регистрируется на сервере MyServer как поступающие от «JohnD» / «Секрет» вместо IUSER_MyServer (который по умолчанию, если включена анонимная проверка подлинности пароля). Аналогично при входе в SQL Server, «JohnD» / «Секрет» используется.  
  
 Следовательно метод проверки подлинности IIS Windows NT вызов-ответ позволяет HTML-страниц, создаваемых без запроса пользователя явным образом для пользователя сведения идентификатор и пароль, необходимые для входа в базу данных. При использовании обычной проверки подлинности IIS были затем это также может потребоваться.  
  
## <a name="password-authentication"></a>Пароль проверки подлинности  
 Служба удаленных рабочих СТОЛОВ может взаимодействовать с веб-сервер IIS под управлением в одном из трех режимов проверки подлинности пароля: Anonymous, Basic или NT вызов-ответ проверки подлинности, в (называемый Windows интегрированной проверки подлинности в Windows 2000). Эти параметры определяют, каким образом веб-сервер контролирует доступ через него, например требование, что клиентские компьютеры имеют явные права на сервере веб-NT.
