# <a name="standard-programming-interface"></a>Стандартный интерфейс программирования
Интерфейс программирования является, пожалуй, наиболее очевидным кандидатом на стандартизацию. На самом деле, когда ODBC разрабатывался, ANSI и ISO уже предоставляли стандарты для встроенных модулей S'L и S'L. Хотя стандартов для базы данных CLI не существует, Группа доступа s'L - отраслевой консорциум поставщиков баз данных - рассматривает вопрос о том, следует ли создать ее; части ODBC более поздно стали основой для их работы.  
  
 Одним из требований к ODBC было то, что одно приложение двоичного должно было работать с несколькими DBMSs. Именно по этой причине ODBC не использует встроенные языки или модульные языки. Несмотря на стандартизируются языки на встроенных языках S-L и модульных языках, каждый из них привязан к прекомпиляторам DBMS. Таким образом, приложения должны быть перекомпилированы для каждого DBMS и в результате бинарные файлы работают только с одной DBMS. Хотя это приемлемо для малообъемных приложений, найденных в миникомпьютерном и мэйнфрейме, это неприемлемо в мире персональных компьютеров. Во-первых, это логистический кошмар, чтобы доставить несколько версий больших объемов, термоусадочных программного обеспечения для клиентов; во-вторых, приложения для персональных компьютеров часто должны одновременно получать доступ к нескольким DBMS.  
  
 С другой стороны, интерфейс уровня вызова может быть реализован через библиотеки или драйверы баз данных, которые находятся на каждой локальной машине; для каждого DBMS требуется другой драйвер. Поскольку современные операционные системы могут загружать такие библиотеки (например, библиотеки динамической связи в операционной системе Microsoft® Windows®) во время выполнения, одно приложение может получить доступ к данным из различных DBMS, а также получить доступ к данным из нескольких баз данных одновременно. По мере появления новых драйверов баз данных пользователи могут просто устанавливать их на свои компьютеры без необходимости изменять, пересоставлять или переукладывать приложения базы данных. Кроме того, интерфейс уровня вызова был хорошим кандидатом для ODBC, потому что Windows - платформа, для которой ODBC был первоначально разработан - уже широко использовал такие библиотеки.
