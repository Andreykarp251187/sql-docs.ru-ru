# <a name="sqlallochandle-function"></a>Функция SQLAllocHandle
**Соответствие стандартам**  
 Представленные версии: ODBC 3.0 стандартов соответствия: ISO-92  
  
 **Сводка**  
 **SQLAllocHandle** выделяет дескриптор среды, подключения, инструкции или дескриптора.  
  
> [!NOTE]  
>  Эта функция является универсальной функции распределения дескрипторов, которая заменяет функции ODBC 2.0 **SQLAllocConnect**, **SQLAllocEnv**, и **SQLAllocStmt**. Чтобы разрешить приложения, вызывающие **SQLAllocHandle** для работы с ODBC 2. *x* драйверы, вызов **SQLAllocHandle** сопоставлен диспетчера драйверов для **SQLAllocConnect**, **SQLAllocEnv**, или  **SQLAllocStmt**, соответствующим образом. Дополнительные сведения см. в разделе «Примечания». Дополнительные сведения о какие диспетчера драйверов сопоставляет эту функцию, чтобы при ODBC 3. *x* при работе с ODBC 2. *x* драйвера, см. в разделе [сопоставление замещающих функций для обеспечения обратной совместимости приложений](../../../odbc/reference/develop-app/mapping-replacement-functions-for-backward-compatibility-of-applications.md).  
  
## <a name="syntax"></a>Синтаксис  
  
```cpp  
  
SQLRETURN SQLAllocHandle(  
      SQLSMALLINT   HandleType,  
      SQLHANDLE     InputHandle,  
      SQLHANDLE *   OutputHandlePtr);  
```  
  
## <a name="arguments"></a>Аргументы  
 *HandleType*  
 [Вход] Тип дескриптора выделения в **SQLAllocHandle**. Должен принимать одно из следующих значений:  
  
-   SQL_HANDLE_DBC  
  
-   SQL_HANDLE_DBC_INFO_TOKEN  
  
-   SQL_HANDLE_DESC  
  
-   SQL_HANDLE_ENV  
  
-   ЗНАЧЕНИЕ SQL_HANDLE_STMT  
  
 Дескриптор SQL_HANDLE_DBC_INFO_TOKEN используется только для диспетчера драйверов и драйверов. Приложения не должны использовать этот тип дескриптора. Дополнительные сведения о SQL_HANDLE_DBC_INFO_TOKEN, см. в разделе [драйвера ODBC с поддержкой пула подключений разработка](../../../odbc/reference/develop-driver/developing-connection-pool-awareness-in-an-odbc-driver.md).  
  
 *InputHandle*  
 [Вход] Дескриптор ввода, в контексте которого является новый маркер для выделения. Если *HandleType* SQL_HANDLE_ENV, — это значение SQL_NULL_HANDLE. Если *HandleType* является SQL_HANDLE_DBC, это должен быть дескриптор среды и если это значение SQL_HANDLE_STMT или SQL_HANDLE_DESC, он должен быть дескриптор соединения.  
  
 *OutputHandlePtr*  
 [Выход] Указатель на буфер, в которую будет возвращен дескриптор в структуру данных во вновь выделенный.  
  
## <a name="returns"></a>Возвращает  
 Значение SQL_SUCCESS, SQL_SUCCESS_WITH_INFO, SQL_INVALID_HANDLE или SQL_ERROR.  
  
 При выделении дескриптор, отличный от дескриптор среды, если **SQLAllocHandle** возвращает значение SQL_ERROR, оно задает *OutputHandlePtr* SQL_NULL_HDBC, SQL_NULL_HSTMT или SQL_NULL_HDESC, в зависимости от значение *HandleType*, если выходной аргумент не является пустым указателем. Приложение может затем получить дополнительные сведения из структуры диагностических данных, связанной с дескриптором в *InputHandle* аргумент.  
  
## <a name="environment-handle-allocation-errors"></a>Ошибки выделения дескриптора среды  
 Распределение среде происходит в пределах диспетчера драйверов и внутри каждого драйвера. Ошибка, возвращенная **SQLAllocHandle** с *HandleType* из SQL_HANDLE_ENV зависит от уровня, в которой произошла ошибка.  
  
 Если диспетчер драйверов не удалось выделить память для  *\*OutputHandlePtr* при **SQLAllocHandle** с *HandleType* SQL_HANDLE_ENV, называется, или приложение предоставляет пустой указатель для *OutputHandlePtr*, **SQLAllocHandle** возвращает ошибку SQL_ERROR. Задает диспетчер драйверов **OutputHandlePtr* для SQL_NULL_HENV (если приложение является пустым указателем, который возвращает значение SQL_ERROR). Нет нет дескриптора, с которым необходимо связать дополнительные диагностические сведения.  
  
 Диспетчер драйверов не вызывает функцию выделения дескриптора среды уровня драйвер пока приложение не вызовет **SQLConnect**, **SQLBrowseConnect**, или **SQLDriverConnect**. При возникновении ошибки на уровне драйвера **SQLAllocHandle** функции, а затем уровень диспетчера драйверов **SQLConnect**, **SQLBrowseConnect**, или  **SQLDriverConnect** функция возвращает значение SQL_ERROR. Структура диагностических данных, содержит SQLSTATE IM004 (драйвера **SQLAllocHandle** сбой). Ошибка возвращена на дескриптор соединения.  
  
 Дополнительные сведения о передаче вызовов функций между диспетчера драйверов и драйверов см. в разделе [функция SQLConnect](../../../odbc/reference/syntax/sqlconnect-function.md).

## <a name="comments"></a>Комментарии  
 **SQLAllocHandle** используется для выделения дескрипторы для сред, соединений, инструкций и дескрипторов, как описано в следующих разделах. Общие сведения о маркерах см. в разделе [дескрипторы](../../../odbc/reference/develop-app/handles.md).  
  
 Более одного дескриптора среды, подключения или инструкции могут быть выделены агентом приложения одновременно, если несколько расположений поддерживаемых драйвером. В ODBC не определено на количество среды, подключения, инструкции или дескриптора, которое может быть выделено в любой момент времени. Драйверы могут накладывает ограничений на количество определенного типа дескриптора, который может быть выделен одновременно; Дополнительные сведения см. в документации драйвера.  
  
 Если приложение вызывает **SQLAllocHandle** с  *\*OutputHandlePtr* значение среды, подключения, инструкции или дескриптора, который уже существует, драйвер перезаписывает сведения, связанные с *обработки*, если приложение не использует подключения регулирования количества запросов (см. в разделе «Выделение среды атрибут для пул соединений» далее в этом разделе). Диспетчер драйверов не проверяет, см. в разделе ли *обрабатывать* в  *\*OutputHandlePtr* уже используется, а также проверьте предыдущее содержимое маркера до их перезаписи .  
  
> [!NOTE]  
>  Это неправильный программирования приложений ODBC для вызова **SQLAllocHandle** два раза с той же переменной приложения, определенные для  *\*OutputHandlePtr* без вызова  **SQLFreeHandle** для освобождения дескриптора перед ее перераспределения. Перезапись ODBC дескрипторы таким образом, может привести к непредсказуемому поведению или ошибок со стороны драйверы ODBC.  
  
 В операционных системах, поддерживающих несколько потоков приложения могут использовать же дескриптор среды, подключения, инструкции или дескриптора в разных потоках. Драйверы должны поддерживать таким образом безопасный, многопоточной доступ к этой информации; один из способов добиться этого, — например, с помощью критической секции или семафора. Дополнительные сведения о работе с потоками см. в разделе [многопоточность](../../../odbc/reference/develop-app/multithreading.md).  
  
 **SQLAllocHandle** не задан атрибут среды SQL_ATTR_ODBC_VERSION при вызове выделить дескриптор среды; атрибуту окружения должен быть установлен приложением, или параметром SQLSTATE HY010 будет (ошибка последовательности функций) возвращается, когда **SQLAllocHandle** вызывается, чтобы выделить дескриптор соединения.  
  
 Для приложений, соответствующих стандартам **SQLAllocHandle** сопоставляется **SQLAllocHandleStd** во время компиляции. Это различие между этими двумя функциями **SQLAllocHandleStd** SQL_ATTR_ODBC_VERSION среды для атрибута устанавливается значение SQL_OV_ODBC3 при вызове с *HandleType* аргумент значение SQL _HANDLE_ENV. Это делается, поскольку соответствующие стандартам приложения всегда ODBC 3. *x* приложений. Кроме того стандарты не требуется версия приложения для регистрации. Это единственное различие между этими двумя функциями; в противном случае они идентичны. **SQLAllocHandleStd** сопоставляется **SQLAllocHandle** внутри диспетчера драйверов. Таким образом, сторонние драйверы не нужно реализовать **SQLAllocHandleStd**.  
  
 Приложения ODBC 3.8, должны использовать:  
  
-   **SQLAllocHandle и не SQLAllocHandleStd** выделить дескриптор среды.  
  
-   **SQLSetEnvAttr** присвоить атрибуту окружения SQL_ATTR_ODBC_VERSION SQL_OV_ODBC3_80.  
  
## <a name="allocating-an-environment-handle"></a>Выделение дескриптора среды  
 Дескриптор среды предоставляет доступ для общих сведений, таких как дескрипторы допустимое подключение и активных дескрипторов соединений. Общие сведения о дескрипторы среды, см. в разделе [дескрипторы среды](../../../odbc/reference/develop-app/environment-handles.md).  
  
 Чтобы запросить дескриптор среды, приложение вызывает **SQLAllocHandle** с *HandleType* из SQL_HANDLE_ENV и *InputHandle* из значение SQL_NULL_HANDLE. Драйвер выделяет память для рабочей среды и передает обратно значение связанного дескриптора  *\*OutputHandlePtr* аргумент. Приложение передает  *\*OutputHandle* значение во всех последующих вызовах, требующих аргумента дескриптор среды. Дополнительные сведения см. в разделе [выделение маркер среды](../../../odbc/reference/develop-app/allocating-the-environment-handle.md).  
  
 В разделе дескриптора среды диспетчера драйверов, если уже существует дескриптора среды драйвера, затем **SQLAllocHandle** с *HandleType* из SQL_HANDLE_ENV не вызывается в этот драйвер при установлено соединение, только **SQLAllocHandle** с *HandleType* из SQL_HANDLE_DBC. Если дескриптор среды драйвера не существует для дескриптора среды диспетчера драйверов, SQLAllocHandle с HandleType SQL_HANDLE_ENV и SQLAllocHandle с HandleType SQL_HANDLE_DBC вызываются в драйвере при первом подключении дескриптор среды подключен к драйверу.  
  
 Когда диспетчер драйверов обрабатывает **SQLAllocHandle** функционировать с *HandleType* из SQL_HANDLE_ENV, он проверяет **трассировки** ключевое слово в разделе [ODBC] системы сведения. Если он имеет значение 1, диспетчер драйверов включает трассировки для текущего приложения. Если флаг трассировки, трассировка запустится при первом дескриптора среды выделяется и заканчивается при освобождении дескриптора среды. Дополнительные сведения см. в разделе [Настройка источников данных](../../../odbc/reference/install/configuring-data-sources.md).  
  
 После выделения дескриптора среды, приложение должно вызвать **SQLSetEnvAttr** на дескриптора среды для установки атрибута SQL_ATTR_ODBC_VERSION среды. Если этот атрибут не задан перед **SQLAllocHandle** вызывается для выделения дескриптора соединения в среде, при вызове установить соединение будет возвращено параметром SQLSTATE HY010 (функционировать ошибка последовательности). Дополнительные сведения см. в разделе [объявление версии ODBC приложения](../../../odbc/reference/develop-app/declaring-the-application-s-odbc-version.md).  
  
## <a name="allocating-shared-environments-for-connection-pooling"></a>Выделение средах общего доступа, для организации пулов соединений  
 Среды могут совместно использоваться несколько компонентов в одном процессе. В то же время совместно используемой среде можно использовать с более чем в одном компоненте. Когда компонент использует совместно используемой среде, его можно использовать в составе пула подключений, что позволяет выделять и использовать существующее соединение без повторного создания этого подключения.  
  
 Прежде чем выделение совместно используемой среде, можно использовать для организации пулов соединений, приложение должно вызвать **SQLSetEnvAttr** присвоить атрибуту окружения SQL_ATTR_CONNECTION_POOLING SQL_CP_ONE_PER_DRIVER или SQL_CP_ONE_PER_ HENV. **SQLSetEnvAttr** в этом случае вызывается с *EnvironmentHandle* значение null, что делает атрибут является атрибутом уровня процесса.  
  
 Когда организация пулов соединений включено, приложение вызывает **SQLAllocHandle** с *HandleType* аргумент значение SQL_HANDLE_ENV. Среды, выделенной с помощью этого вызова будет неявное общей среды, так как пул подключений включен.  
  
 При выделении общей среде, среде, которая будет использоваться будет неопределенным до **SQLAllocHandle** с *HandleType* SQL_HANDLE_DBC, называется. После этого диспетчер драйверов пытается найти существующую среду, соответствующую атрибуты среды, запрашиваемой приложением. Если среда не существует, он создается как совместно используемой среде. Диспетчер драйверов поддерживает счетчик ссылок для каждой общей среды; Счетчик имеет значение 1, при создании среды. Если обнаруживается соответствующая среда, в приложение возвращается дескриптор этой среды и счетчик ссылок увеличивается. Дескриптор среды, выделенной таким образом можно использовать в любой функции ODBC, который принимает дескриптор среды в качестве входного аргумента.  
  
## <a name="allocating-a-connection-handle"></a>Выделение дескриптора соединения  
 Дескриптор соединения предоставляет доступ к информации, такие как допустимую инструкцию и откройте указатели дескрипторов на подключение и является ли транзакция в данный момент. Общие сведения о дескрипторов соединений, см. в разделе [дескрипторов соединений](../../../odbc/reference/develop-app/connection-handles.md).  
  
 Чтобы запросить дескриптор соединения, приложение вызывает **SQLAllocHandle** с *HandleType* из SQL_HANDLE_DBC. *InputHandle* аргумент имеет значение дескриптора среды, который был возвращен вызовом **SQLAllocHandle** , которые выделены этот дескриптор. Драйвер выделяет память для сведений о соединении и передает обратно значение связанного дескриптора  *\*OutputHandlePtr*. Приложение передает  *\*OutputHandlePtr* значение во всех последующих вызовах, требующими дескриптор соединения. Дополнительные сведения см. в разделе [выделить дескриптор соединения](../../../odbc/reference/develop-app/allocating-a-connection-handle-odbc.md).  
  
 Обрабатывает диспетчера драйверов **SQLAllocHandle** функции и вызывает драйвер **SQLAllocHandle** функционировать, когда приложение вызывает **SQLConnect**, **SQLBrowseConnect**, или **SQLDriverConnect**. (Дополнительные сведения см. в разделе [функция SQLConnect](../../../odbc/reference/syntax/sqlconnect-function.md).)  
  
 Если атрибут SQL_ATTR_ODBC_VERSION среды не задана до **SQLAllocHandle** вызывается для выделения дескриптора соединения в среде, при вызове установить соединение будет возвращено параметром SQLSTATE HY010 (функционировать последовательности Ошибка).  
  
 Если приложение вызывает **SQLAllocHandle** с *InputHandle* аргумента значение SQL_HANDLE_DBC, а также задать в общей среде дескриптор, диспетчер драйверов пытается найти существующее общее Среда, которая соответствует атрибуты среды, установленный приложением. Если среда не существует, он создается, с числом ссылок (поддерживается диспетчером драйверов) 1. Если соответствие общим найти среду, этот дескриптор возвращается в приложение и его счетчик ссылок увеличивается.  
  
 Фактическое подключение, который будет использоваться не определяется диспетчером драйверов до **SQLConnect** или **SQLDriverConnect** вызывается. Диспетчер драйверов использует параметры подключения в вызове **SQLConnect** (или ключевых слов подключения в вызове **SQLDriverConnect**) и задать атрибуты соединения после выделения подключения Определите, какие соединения в пуле должен использоваться. Дополнительные сведения см. в разделе [функция SQLConnect](../../../odbc/reference/syntax/sqlconnect-function.md).  
  
## <a name="allocating-a-statement-handle"></a>Выделение дескриптора инструкции  
 Дескриптор инструкции предоставляет доступ к сведениям инструкции, такие как сообщения об ошибках, имя курсора и сведения о состоянии для обработки инструкций SQL. Общие сведения о дескрипторов инструкций, см. в разделе [дескрипторов инструкций](../../../odbc/reference/develop-app/statement-handles.md).  
  
 Чтобы запросить дескриптор инструкции, приложение подключается к источнику данных, а затем вызывает **SQLAllocHandle** прежде чем отправляет инструкции SQL. В этом вызове *HandleType* должно быть присвоено значение SQL_HANDLE_STMT и *InputHandle* должно быть присвоено дескриптора соединения, который был возвращен вызовом **SQLAllocHandle** которые выделены этот дескриптор. Драйвер выделяет память для данных инструкции, связывает дескриптора инструкции с указанным соединением и передает обратно значение связанного дескриптора  *\*OutputHandlePtr*. Приложение передает  *\*OutputHandlePtr* значение во всех последующих вызовах, требующими дескриптор инструкции. Дополнительные сведения см. в разделе [выделить дескриптор инструкции](../../../odbc/reference/develop-app/allocating-a-statement-handle-odbc.md).  
  
 Когда выделяется дескриптор инструкции, драйвер автоматически выделяет набор четырех дескрипторы и назначает дескрипторы для этих дескрипторов SQL_ATTR_APP_ROW_DESC, SQL_ATTR_APP_PARAM_DESC, SQL_ATTR_IMP_ROW_DESC и SQL_ATTR_IMP_PARAM_DESC атрибуты инструкции. Они называются *неявно* выделенных дескрипторами. Чтобы явно выделить дескриптор приложений, обратитесь к следующему разделу, «Выделение дескриптора».  
  
## <a name="allocating-a-descriptor-handle"></a>Выделение дескриптора дескриптора  
 Если приложение вызывает **SQLAllocHandle** с *HandleType* из SQL_HANDLE_DESC, драйвер выделяет дескриптор приложений. Они называются *явно* выделенных дескрипторами. Приложение направляет драйверы, чтобы воспользоваться дескриптор явно выделенные приложений вместо одной из автоматически выделенный для данной инструкции дескриптор, вызвав **SQLSetStmtAttr** функцию с SQL_ATTR_APP_ROW_DESC или атрибут SQL_ATTR_APP_PARAM_DESC. Явным образом не удалось выделить дескриптор реализации, а также можно указать дескриптор реализации в **SQLSetStmtAttr** вызов функции.  
  
 Явно выделенные дескрипторы связаны с дескриптор соединения вместо дескриптора инструкции (как автоматически выделенные дескрипторы). Дескрипторы остаются выделенными, только в том случае, если приложение фактически подключено к базе данных. Поскольку явно выделенные дескрипторы, связанные с дескриптор соединения, приложение можно связать дескриптор явно выделенные с более одной инструкции в подключении. С другой стороны, дескриптор неявно выделенные приложений не может быть связан более одного дескриптора инструкции. (Он не может быть связан с любой дескриптор инструкции, отличном от того, в который он был выделен для.) Явно выделенные указатели дескрипторов можно явно освободить приложение или вызвав **SQLFreeHandle** с *HandleType* SQL_HANDLE_DESC или неявно при подключении Закрыто.  
  
 Когда освобождается явно выделенные дескриптора, неявно выделенные дескриптор снова связан с инструкцией. (Атрибут SQL_ATTR_APP_ROW_DESC или SQL_ATTR_APP_PARAM_DESC для этой инструкции будет снова присвоено неявно выделенного дескриптора.) Это справедливо для всех инструкций, которые были связаны с явно выделенные дескриптором для соединения.  
  
 Дополнительные сведения о дескрипторах, см. в разделе [дескрипторы](../../../odbc/reference/develop-app/descriptors.md).
