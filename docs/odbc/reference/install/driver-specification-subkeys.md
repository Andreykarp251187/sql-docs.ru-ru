# <a name="driver-specification-subkeys"></a>Подразделы спецификаций драйверов
Каждый драйвер, перечисленных в подразделе драйверы ODBC содержит подраздел свои собственные. Этот подраздел содержит имя, совпадающее с именем соответствующего значения в подразделе драйверы ODBC. Значения в этом подразделе перечислены полные пути драйвера и установка драйверов GPU библиотеки DLL, значения ключевых слов драйвера, возвращенный **SQLDrivers**и счетчик использования. Как показано в следующей таблице, используются форматы значений.  
  
|Name|Тип данных|Data|  
|----------|---------------|----------|  
|APILevel|REG_SZ|**0** &#124; **1** &#124; **2**|  
|ConnectFunctions|REG_SZ|{**Y**&#124;**N**} {**Y**&#124;**N**} {**Y**&#124;**N**}|  
|CreateDSN|REG_SZ|*Описание драйвера*|  
|Драйвер|REG_SZ|*путь к библиотеке DLL драйвера*|  
|DriverODBCVer|REG_SZ|*nn.nn*|  
|FileExtns|REG_SZ|**\*.** *файл extension1*[ **,\*.** *файл extension2*]...|  
|FileUsage|REG_SZ|**0** &#124; **1** &#124; **2**|  
|Установка|REG_SZ|*путь к библиотеке DLL программы установки*|  
|SQLLevel|REG_SZ|**0** &#124; **1** &#124; **2**|  
|UsageCount|REG_DWORD|*count*|  
  
 В следующей таблице показано использование каждого ключевого слова.  
  
|Ключевое слово|Использование|  
|-------------|-----------|  
|**APILevel**|Число, указывающее ODBC интерфейса уровень согласованности, поддерживаемых драйвером:<br /><br /> 0 = нет<br /><br /> 1 = поддерживается уровня 1<br /><br /> 2 = поддерживается уровня 2<br /><br /> Это должен быть таким же, как значение, возвращаемое для параметра SQL_ODBC_INTERFACE_CONFORMANCE в **SQLGetInfo**.|  
|**CreateDSN**|Имя одного или нескольких источников данных, создаваемого при установке драйвера. Сведения о системе должен содержать один раздел спецификации источника данных для каждого источника данных, в списке **CreateDSN** ключевое слово. Эти разделы не следует включать **драйвер** ключевое слово, так как это указано в разделе спецификации драйвер, но должен содержать достаточно информации для **ConfigDSN** функции в драйвере библиотеки DLL, создание спецификации источника данных без отображения все диалоговые окна программы установки. Формат раздела спецификации источника данных, см. в разделе [подразделы спецификации источника данных](../../../odbc/reference/install/data-source-specification-subkeys.md).|  
|**ConnectFunctions**|Строка трех символов, указывающее, поддерживает ли драйвер **SQLConnect**, **SQLDriverConnect**, и **SQLBrowseConnect**. Если драйвер поддерживает **SQLConnect**, первый символ — «Y»; в противном случае это «N». Если драйвер поддерживает **SQLDriverConnect**, второй символ равен «Y»; в противном случае это «N». Если драйвер поддерживает **SQLBrowseConnect**, третьего знака — «Y»; в противном случае это «N». Например, если драйвер поддерживает **SQLConnect** и **SQLDriverConnect** , но не **SQLBrowseConnect**, строка трех знаков является «YYN».|  
|**DriverODBCVer**|Строка символов, с помощью версии ODBC, который поддерживает драйвер. Версия имеет форму *nn.nn*, где первые две цифры являются основной номер версии, а две следующие — дополнительный номер версии. Для версии ODBC, описанные в данном руководстве драйвер должен возвращать «03.00».<br /><br /> Это должен быть таким же, как значение, возвращаемое для параметра SQL_DRIVER_ODBC_VER в **SQLGetInfo**.|  
|**FileExtns**|Для драйверов на основе файла разделенный запятыми список расширений файлов драйвер можно использовать. Например, указать драйвер для dBASE \*.dbf и драйвер файловой форматированного текста может указать \*.txt,\*CSV-файл. Пример того, как приложение может использовать эти сведения, см. в разделе **FileUsage** ключевое слово.|  
|**FileUsage**|Число, указывающее, каким образом драйверов на основе файла напрямую обрабатывает файлы в источнике данных.<br /><br /> 0 = этот драйвер не драйверов на основе файла. Например драйвер ORACLE — это драйвер основе СУБД.<br /><br /> 1 = файлы рассматривает драйверов на основе файла в источнике данных как таблицы. Например драйвером Xbase считает каждый файл Xbase таблицы.<br /><br /> 2 = файлов драйверов на основе файла рассматривает в качестве каталога источника данных. Например драйвер Microsoft® Access обрабатывает каждый файл Microsoft Access как всей базы данных.<br /><br /> Приложение может использовать это, чтобы определить, каким образом пользователи будут выбирать данные. Например Xbase и Paradox пользователи часто прибегают к данных, хранящиеся в файлах, хотя ORACLE и Microsoft Access пользователи обычно думают о данных, хранящихся в таблицах.<br /><br /> Когда пользователь выбирает **Открытие файла данных** из **файл** меню, приложение может выводить **Открытие файла Windows** общее диалоговое окно. Список типов файлов будет использовать указанные расширения файла с **FileExtns** ключевое слово для драйверов, которые указывают **FileUsage** значение 1 и «Y», как второй знак значения  **ConnectFunctions** ключевое слово. После того как пользователь выберет файл, приложение может вызвать **SQLDriverConnect** с **ДРАЙВЕР** ключевое слово, а затем выполните **ВЫБЕРИТЕ \* FROM *имя таблицы***   инструкции.<br /><br /> Когда пользователь выбирает **импорта данных** из **файл** меню, приложение может выводить список описаний для драйверов, которые указывают **FileUsage** значение 0 или 2 и «Y» как второй знак значения **ConnectFunctions** ключевое слово. После того как пользователь выберет драйвером, приложение может вызвать **SQLDriverConnect** с **ДРАЙВЕР** ключевое слово, а затем отобразите пользовательский **Выбор таблицы** диалоговое окно.|  
|**SQLLevel**|Число, указывающее грамматику SQL-92, поддерживаемых драйвером:<br /><br /> 0 = операция SQL-92<br /><br /> 1 = FIPS127-2 Transitional<br /><br /> 2 = промежуточного SQL-92<br /><br /> 3 = Полный SQL-92<br /><br /> Это должен быть таким же, как значение, возвращаемое для параметра SQL_SQL_CONFORMANCE в **SQLGetInfo**.|  
  
 Сведения о счетчиках использования см. в разделе [подсчет использования](../../../odbc/reference/install/usage-counting.md) ранее в этом разделе.  
  
 Приложения, не задавайте счетчик использования. ODBC будет поддерживать этот счетчик.  
  
 Например предположим, что драйвер для форматированного текста файлов включает драйвер, библиотеки DLL с именем Text.dll, установки отдельного драйвера библиотеки DLL с именем Txtsetup.dll и установлена в три раза. Если драйвер поддерживает уровень API 1 уровень соответствия, поддерживает уровень соответствия грамматики SQL минимум, обрабатывает файлы как таблицы и файлы можно использовать с расширениями txt- и CSV-файл, значения в подразделе текста может выглядеть следующим образом:  
  
```  
APILevel : REG_SZ : 1  
ConnectFunctions : REG_SZ : YYN  
Driver : REG_SZ : C:\WINDOWS\SYSTEM32\TEXT.DLL  
DriverODBCVer : REG_SZ : 03.00.00  
FileExtns : REG_SZ : *.txt,*.csv  
FileUsage : REG_SZ : 1  
Setup : REG_SZ : C:\WINDOWS\SYSTEM32\TXTSETUP.DLL  
SQLLevel : REG_SZ : 0  
UsageCount : REG_DWORD : 0x3  
```
