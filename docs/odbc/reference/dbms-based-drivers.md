# <a name="dbms-based-drivers"></a>Драйверы на основе СУБД
Драйверы на основе DBMS используются с источниками данных, такими как Oracle или S'L Server, которые обеспечивают автономный движок базы данных для водителя. Эти драйверы получают доступ к физическим данным через автономный движок; т.е. они отправляют заявления s'L и извлекают результаты из движка.  
  
 Поскольку драйверы на основе DBMS используют существующий движок базы данных, их обычно легче писать, чем драйверы на основе файлов. Хотя драйвер на основе DBMS может быть легко реализован путем перевода вызовов ODBC на родные вызовы API, это приводит к более медленному драйверу. Лучший способ реализации драйвера на основе DBMS — использовать базовый протокол потока данных, что обычно является тем, чем занимается родной API. Например, драйвер сервера S'L должен использовать TDS (протокол потока данных для сервера S'L), а не DB Library (родной API для сервера S'L). Исключением из этого правила является то, что ODBC является родным API. Например, Watcom S'L является автономным двигателем, который находится на той же машине, что и приложение, и загружается непосредственно в качестве драйвера.  
  
 Драйверы на основе DBMS выступают в качестве клиента в конфигурации клиента/сервера, где источник данных выступает в качестве сервера. В большинстве случаев клиент (водитель) и сервер (источник данных) находятся на разных машинах, хотя оба могут проживать на одной и той же машине, работающей с многозадачной операционной системой. Третьей возможностью является *шлюз,* который находится между драйвером и источником данных. Шлюз — это часть программного обеспечения, которая заставляет один DBMS выглядеть как другой. Например, приложения, написанные для использования S'L Server, также могут получить доступ к данным DB2 через шлюз Micro Decisionware DB2; этот продукт приводит к тому, что DB2 выглядит как сервер S'L.  
  
 На следующей иллюстрации показаны три различные конфигурации драйверов на основе DBMS. В первой конфигурации драйвер и источник данных находятся на одной машине. Во втором, драйвер и источник данных находятся на разных машинах. В третьем, водитель и источник данных находятся на разных машинах, и шлюз сидит между ними, проживая на еще одной машине.
