# <a name="concurrency-types"></a>Типы параллелизма
Чтобы решить проблему снижения параллелизма в курсорах, ODBC предоставляет четыре различных типа курсора:  
  
-   **Только для чтения** Курсор может считывать данные, но не может обновлять или удалять данные. Это тип парвалюты по умолчанию. Хотя DBMS может блокировать строки для обеспечения повторного чтения и serializable уровней изоляции, он может использовать чтения замки вместо записи замков. Это приводит к более высокой параллелизме, поскольку другие транзакции могут по крайней мере читать данные.  
  
-   **Блокировка** Курсор использует самый низкий уровень блокировки, необходимый для того, чтобы он мог обновлять или удалять строки в наборе результатов. Обычно это приводит к очень низким уровням параллелизма, особенно на уровнях изоляции транзакций Повторяемого чтения и serializable.  
  
-   **Оптимистический параллелизм с использованием строковых версий и оптимистичного параллелизма с использованием значений** Курсор использует оптимистический параллелизм: он обновляет или удаляет строки, только если они не изменились, так как они были последним чтением. Для обнаружения изменений сравниваются строки версий или значений. Нет никакой гарантии, что курсор сможет обновить или удалить строку, но параллелизм намного выше, чем при использовании блокировки. Для получения дополнительной информации, [Optimistic Concurrency](../../../odbc/reference/develop-app/optimistic-concurrency.md)см.  
  
 Приложение определяет, какой тип параллелизма он хочет использовать курсор с атрибутом SQL_ATTR_CONCURRENCY оператора. Чтобы определить, какие типы поддерживаются, он называет **s'LGetInfo** с SQL_SCROLL_CONCURRENCY опцией.
