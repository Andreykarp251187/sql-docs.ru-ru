# <a name="core-interface-conformance"></a>Соответствие основного интерфейса
Все драйверы ODBC должны обладать по крайней мере уровень ядра соответствие интерфейса. Так как функции на уровне ядра, предусмотренные стандартом наиболее универсальных приложений с возможностью взаимодействия, драйвер может работать с таких приложений. Возможности уровня ядра также соответствуют для функций, определенных в спецификации ISO CLI и необходимые компоненты, определенные в спецификации Open CLI группы. Драйвер ODBC интерфейс совместимого уровня ядра позволяет приложению выполнять любые из следующих:  
  
-   Выделить и освободить все типы дескрипторов, путем вызова метода **SQLAllocHandle** и **SQLFreeHandle**.  
  
-   Использовать все виды **SQLFreeStmt** функции.  
  
-   Привязать столбцы результирующего набора, путем вызова **SQLBindCol**.  
  
-   Обрабатывать динамические параметры, включая массивы параметров ввода только в направлении, путем вызова **SQLBindParameter** и **SQLNumParams**. (Параметры в направлении выходные данные являются компонентов 203 в [соответствие интерфейса уровня 2](../../../odbc/reference/develop-app/level-2-interface-conformance.md).)  
  
-   Укажите смещение привязки.  
  
-   Воспользуйтесь диалоговым окном данных во время выполнения, включающие вызовы **SQLParamData** и **SQLPutData**.  
  
-   Управление курсоры и курсоры, путем вызова **SQLCloseCursor**, **SQLGetCursorName**, и **SQLSetCursorName**.  
  
-   Получить доступ к описанию (метаданные) результирующих наборов, путем вызова **SQLColAttribute**, **SQLDescribeCol**, **SQLNumResultCols**, и **SQLRowCount** . (Использование этих функций на номер столбца 0 для получения метаданных закладки является функция 204 в [соответствие интерфейса уровня 2](../../../odbc/reference/develop-app/level-2-interface-conformance.md).)  
  
-   Запрос данных словаря, путем вызова функций каталога **SQLColumns**, **SQLGetTypeInfo**, **SQLStatistics**, и **SQLTables**.  
  
     Драйвер не является обязательным для поддержки многокомпонентных именах таблиц базы данных и представлений. (Дополнительные сведения см. в разделе функция 101 [соответствие интерфейса уровня 1](../../../odbc/reference/develop-app/level-1-interface-conformance.md) и 201 в [соответствие интерфейса уровня 2](../../../odbc/reference/develop-app/level-2-interface-conformance.md).) Тем не менее некоторые функции спецификации SQL-92, например столбец квалификации и имена индексов, синтаксически сравнимы составного именования. Присутствует список функций ODBC не предназначен для введения новых параметров в эти аспекты SQL-92.  
  
-   Управление источниками данных и подключениях, путем вызова **SQLConnect**, **SQLDataSources**, **SQLDisconnect**, и **SQLDriverConnect**. Получить сведения о драйверах, независимо от того, какие ODBC уровня они поддерживают, путем вызова **SQLDrivers**.  
  
-   Подготовка и выполнение инструкций SQL, путем вызова **SQLExecDirect**, **SQLExecute**, и **SQLPrepare**.  
  
-   Получить, вызвав одну строку результирующего набора или несколько строк, в прямом направлении, **SQLFetch** или вызвав **SQLFetchScroll** с *FetchOrientation* аргумент Задайте для SQL_FETCH_NEXT.  
  
-   Получить несвязанного столбца в части, с помощью функции **SQLGetData**.  
  
-   Получить текущие значения всех атрибутов, вызвав **SQLGetConnectAttr**, **SQLGetEnvAttr**, и **SQLGetStmtAttr**и установить все атрибуты к значениям по умолчанию и пометить некоторые из атрибутов для нестандартные значения путем вызова **SQLSetConnectAttr**, **SQLSetEnvAttr**, и **SQLSetStmtAttr**.  
  
-   Управлять некоторые поля дескрипторов, путем вызова **SQLCopyDesc**, **SQLGetDescField**, **SQLGetDescRec**, **SQLSetDescField**, и **SQLSetDescRec**.  
  
-   Диагностические сведения, получить, вызвав **SQLGetDiagField** и **SQLGetDiagRec**.  
  
-   Обнаруживать возможности драйвера, путем вызова **SQLGetFunctions** и **SQLGetInfo**. Кроме того, определить результат любой текст подстановками инструкцию SQL перед их отправкой к источнику данных, вызвав **SQLNativeSql**.  
  
-   Используйте синтаксис **SQLEndTran** зафиксировать транзакцию. Драйвер уровня ядра не обязательно должен поддерживать значение true, транзакции. Таким образом для атрибута соединения SQL_ATTR_AUTOCOMMIT приложения нельзя указать SQL_ROLLBACK ни SQL_AUTOCOMMIT_OFF. (Дополнительные сведения см. в разделе функция 109 в [соответствие интерфейса уровня 2](../../../odbc/reference/develop-app/level-2-interface-conformance.md).)  
  
-   Вызовите **SQLCancel** для отмены диалогового окна данных во время выполнения и в многопоточной среде, для отмены выполнения функции ODBC в другом потоке. Соответствие интерфейса уровня ядра не предусматривает поддержку асинхронное выполнение функций, а также использование **SQLCancel** отменить асинхронное выполнение функции ODBC. Не платформы, и драйвер ODBC не обязательно должен быть многопоточных драйвер провести независимых действий одновременно. Тем не менее в многопоточной среде, драйвер ODBC должен быть поточно ориентированными. Сериализация запросов из приложения — способ реализации этой спецификации, соответствующие стандарту, несмотря на то, что он может создать значительное снижение производительности.  
  
-   Получить, вызвав таблиц, столбец идентификации строки SQL_BEST_ROWID **SQLSpecialColumns**. (Поддержка SQL_ROWVER является функция 208 в [соответствие интерфейса уровня 2](../../../odbc/reference/develop-app/level-2-interface-conformance.md).)  
  
    > [!IMPORTANT]  
    >  Драйверы ODBC необходимо реализовать функции уровень соответствия интерфейс Core.
