# <a name="processing-a-sql-statement"></a>Обработка инструкции SQL
Прежде чем обсуждать методы использования S'L программно, необходимо обсудить, как обрабатывается выписка s'L. Шаги, участвующие являются общими для всех трех методов, хотя каждый метод выполняет их в разное время. На следующей иллюстрации показаны шаги, связанные с обработкой выписки по S'L, которые обсуждаются на протяжении всего этого раздела.  
  
 ![Шаги для обработки инструкции SQL](../../odbc/reference/media/pr01.gif "pr01")  
  
 Для обработки оператора S'L DBMS выполняет следующие пять шагов:  
  
1.  DBMS сначала разбирает заявление S'L. Он разбивает заявление на отдельные слова, называемые токенами, гарантирует, что заявление имеет действительный глагол и действительные положения, и так далее. Ошибки и опечатки syntax могут быть обнаружены на этом этапе.  
  
2.  DBMS проверяет заявление. Он проверяет выписку в отношении каталога системы. Все ли таблицы, указанные в выписке, существуют в базе данных? Все ли столбцы существуют и являются ли названия столбцов однозначными? Есть ли у пользователя необходимые привилегии для выполнения оператора? На этом этапе могут быть обнаружены некоторые семантические ошибки.  
  
3.  DBMS генерирует план доступа к выписке. План доступа представляет собой двоичное представление шагов, необходимых для выполнения оператора; это dBMS эквивалент исполняемого кода.  
  
4.  DBMS оптимизирует план доступа. В нем рассматриваются различные способы выполнения плана доступа. Можно ли использовать индекс для ускорения поиска? Должен ли DBMS сначала применить условие поиска к таблице А, а затем присоединиться к таблице B, или же он должен начаться с соединения и использования состояния поиска после этого? Можно ли избежать последовательного поиска через таблицу или свести к подмножеству таблицы? Изучив альтернативы, DBMS выбирает одну из них.  
  
5.  DBMS выполняет выписку, выполняя план доступа.  
  
 Шаги, используемые для обработки оператора S'L, различаются по объему требуемого доступа к базе данных и количеству времени, которое они займут. Анализ оператора S'L не требует доступа к базе данных и может быть выполнен очень быстро. Оптимизация, с другой стороны, очень процессор-интенсивный процесс и требует доступа к каталогу системы. Для сложного многозадачного запроса оптимизатор может исследовать тысячи различных способов выполнения одного и того же запроса. Однако затраты на выполнение запроса неэффективно, как правило, настолько высоки, что время, затрачиваемые на оптимизацию, более чем восстановлено в увеличенной скорости выполнения запроса. Это еще более важно, если один и тот же оптимизированный план доступа можно использовать снова и снова для выполнения повторяющихся запросов.
