# <a name="driver-architecture"></a>Архитектура драйвера
Архитектура драйвера делится на две категории, в зависимости от того, какое программное обеспечение обрабатывает операторы S'L:  
  
-   **Драйверы на основе файлов** Водитель получает прямой доступ к физическим данным. В этом случае водитель выступает как драйвер, так и источник данных; то есть, он обрабатывает вызовы ODBC и операторы S'L. Например, драйверы dBASE являются драйверами на основе файлов, поскольку dBASE не предоставляет автономный движок базы данных, который может использовать водитель. Важно отметить, что разработчики драйверов файлов должны писать свои собственные движки баз данных.  
  
-   **Драйверы на основе DBMS** Водитель получает доступ к физическим данным через отдельный движок базы данных. В этом случае драйвер обрабатывает только вызовы ODBC; он передает операторские заявления в движок базы данных для обработки. Например, драйверы Oracle — это драйверы на основе DBMS, поскольку Oracle имеет автономный движок базы данных, который использует водитель. В тех случаях, когда находится движок базы данных, не имеет значения. Он может распорядиться на той же машине, что и водитель, или другой машиной в сети; к нему может быть даже доступ через шлюз.  
  
 Архитектура драйвера, как правило, интересна только авторам драйверов; то есть архитектура драйвера обычно не имеет никакого значения для приложения. Тем не менее, архитектура может повлиять на то, может ли приложение использовать dBMS-специфический S'L. Например, Microsoft Access предоставляет автономный движок баз данных. Если драйвер Microsoft Access основан на DBMS - он получает доступ к данным через этот движок - приложение может передавать операторы Microsoft Access-S'L в движок для обработки.  
  
 Однако, если драйвер основан на файлах, т.е. содержит запатентованную движок, которая получает доступ к файлу Microsoft® Access .mdb напрямую, любые попытки передать движок специфических инструкций Microsoft Access, специфических S'L, могут привести к ошибкам синтаксиса. Причина в том, что собственный движок, скорее всего, будет реализовывать только ODBC S'L.
