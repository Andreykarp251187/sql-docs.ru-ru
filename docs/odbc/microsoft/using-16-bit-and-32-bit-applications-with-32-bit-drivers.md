# <a name="using-16-bit-and-32-bit-applications-with-32-bit-drivers"></a>Использование 16-разрядных и 32-разрядных приложений с 32-разрядными драйверами
> [!IMPORTANT]  
>  поддержка 16-разрядных приложений будет исключена в будущих версиях Windows. Избегайте использования этого компонента в новых разработках и запланируйте изменение существующих приложений, в которых он применяется. Вместо этого Разрабатывайте 32-разрядные или 64-разрядные приложения.  
  
 С помощью компонента доступа к данным ODBC можно использовать 16-разрядные и 32-разрядные приложения с 32-разрядными драйверами. Операционные системы Microsoft® Windows® 95/98 и Microsoft Windows NT®/Виндовс 2000 поддерживают следующие сочетания приложений и драйверов:  
  
-   16-разрядные приложения с 32-разрядными драйверами  
  
-   32-разрядные приложения с 32-разрядными драйверами  
  
 Использование 32-разрядного приложения с 16-разрядным драйвером не поддерживается.  
  
> [!NOTE]  
>  Начиная с версии ODBC 3,0, поддерживается Windows NT 4,0.  
  
 ODBC включает компоненты ODBC, необходимые для поддержки описанных выше конфигураций с помощью «преобразователей» библиотек динамической компоновки (DLL) для преобразования 16-разрядных адресов в 32-разрядные адреса и наоборот. Программа установки определяет, какая операционная система используется, и устанавливает компоненты ODBC, необходимые для этой системы. Также можно выбрать установку компонентов ODBC, используемых всеми системами.  
  
 В большинстве случаев перенос приложения или драйвера с 16-разрядной на 32-bit включает в себя пять типов изменений:  
  
-   Изменения в коде обработки сообщений  
  
-   Изменения, так как целые числа и дескрипторы имеют 32 бит  
  
-   Изменения в вызовах программных интерфейсов Windows (API)  
  
-   Изменения, обеспечивающие потокобезопасность драйвера  
  
-   Изменения в компонентах ODBC  
  
 С точки зрения программирования приложений или драйверов основное различие между 16-и 32-разрядными компонентами ODBC заключается в том, что они имеют разные имена файлов. С точки зрения системы архитектура каждого подключения к приложению или драйверу отличается, и средства, используемые для управления источниками данных, отличаются.
