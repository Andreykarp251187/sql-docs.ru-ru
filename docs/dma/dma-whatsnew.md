# <a name="whats-new-in-data-migration-assistant"></a>Новые возможности Data Migration Assistant
В этой статье представлены сведения о материалах в каждом выпуске из Data Migration Assistant (DMA).

## <a name="dma-v41"></a>Версии 4.1 DMA
В выпуске версии 4.1 DMA введена поддержка комплексной оценки локальных баз данных SQL Server переход на базу данных управляемого экземпляра SQL Azure.

Рабочий процесс оценки поможет вам обнаружить следующие проблемы, которые могут повлиять на миграцию в управляемый экземпляр Azure SQL Database:

- **Неподдерживаемый или частично поддерживаемые функции**. DMA оценивает базы данных SQL Server источника для возможности использования, которые частично поддерживаются и не поддерживается в целевой базе данных управляемого экземпляра SQL Azure. Затем средство предоставляет полный набор рекомендаций, альтернативные подходы, доступные в Azure и выполнения исправлений шаги, благодаря чему клиенты может использовать эту информацию в учетной записи, при планировании своих проектов миграции.

- **Проблемы совместимости**. DMA также определяет проблемы совместимости, связанные со следующими областями:

    - Критически важные изменения.  Определенной схеме объекты, которые могут нарушить работу, переход на целевую базу данных.  Мы рекомендуем исправления этих объектов схемы после миграции базы данных.
    - Изменения в поведении: Объекты схемы сообщил могут продолжать работать, но они могут так себя вести иное поведение, например снижение производительности.
    - Информационное проблемы:  Эти объекты не влияют на миграцию, но может рекомендованных к использованию функции выпусков SQL Server.

После завершения оценки с помощью наших [Azure Database Migration Service](https://azure.microsoft.com/services/database-migration/) (DMS), чтобы выполнить миграцию баз данных SQL Server для базы данных управляемого экземпляра SQL Azure.  DMS поддерживает как [автономной](https://docs.microsoft.com/azure/dms/tutorial-sql-server-to-managed-instance) (однократный) и [online](https://docs.microsoft.com/azure/dms/tutorial-sql-server-managed-instance-online) перенос баз данных (минимальным временем простоя) базы данных управляемого экземпляра SQL Azure.

## <a name="dma-v40"></a>DMA v4.0
В выпуске версии 4.0 пакета DMA представлены функцию рекомендации номер SKU базы данных SQL Azure, которая позволяет пользователям указывать минимальный рекомендуемый номер SKU базы данных SQL Azure на основе счетчиков производительности, собранных с компьютеров, на котором размещены ваши базы данных. Этот компонент предоставляет рекомендации, связанные с цены на уровень, уровень вычислений и максимальный объем данных, а также оценочная стоимость в месяц. Он также предлагает возможность подготавливать всех баз данных в Azure в пакетном режиме.

> [!NOTE]
> Эта функция является в настоящее время доступны только с помощью интерфейса командной строки (CLI). Поддержка этой функции через пользовательский интерфейс DMA запланирована для доставки позднее в этом году.

Дополнительные сведения см. в статье [определить нужный номер SKU базы данных Azure SQL для базы данных в локальной](dma-sku-recommend-sql-db.md).

## <a name="dma-v36"></a>DMA v3.6
В выпуске v3.6 пакета DMA представлены «Автоматическое исправление» для объектов схемы, на которые влияет наиболее распространенных проблем, препятствующих миграции.

В этом выпуске представлены autofix для следующих проблема, препятствующая миграции и поведение изменяется проблемы:
- Объекты схемы, которые используют синтаксис неполное соединение.
- Объекты схемы, использующие устаревшие инструкции RAISEERROR.
- Инструкции SQL, использующие порядок по целочисленный литерал.

DMA выполняет схемы автоматического преобразования для объектов, затрагиваемых перечисленных проблемах и запрашивает у пользователя подтверждение перед выполнением преобразования схемы. Пользователей можно просмотреть изменения предложенный код и затем либо принять или отклонить все преобразования для любого объекта в данной базе данных.

DMA использует технологию синтеза программы Майкрософт (PROSE), чтобы предложить средства исправления кода. Дополнительные сведения о [PROSE](https://microsoft.github.io/prose/).

## <a name="dma-v35"></a>Версии 3.5 DMA
В выпуске версии 3.5 DMA дополнениями:
- Значительное повышение производительности по переносу базы данных SQL Azure (benchmark результатам тестирования выяснится, что процесс выполняется в четыре раза быстрее, чем в предыдущих версиях DMA).
- Объем памяти Дополнительно оптимизирован для повышения стабильности рабочего процесса миграции.
- Возможность пропускать оценки во время миграции схемы и данных (Если вы уже выполнена оценка и устранить любые критические объекты схемы, перед миграцией).
- Исправление для устранения проблемы, с помощью средства аварийного завершения Недопустимый сетевой путь общего ресурса предоставляется для файлов резервных копий при выполнении обновления предыдущих версий SQL Server в локальной до более поздней версии или в SQL Server на виртуальных машинах Azure.

## <a name="dma-v34"></a>Версии 3.4 DMA
В выпуске версии 3.4 DMA дополнениями:
- Поддержка SQL Server 2017 в качестве источника для миграции в базу данных SQL Azure.
- Улучшения надежности, производительности и оценки правильности правило.

## <a name="dma-v33"></a>Версии 3.3 DMA
В выпуске версии 3.3 DMA обеспечивает миграцию с локальным экземпляром SQL Server до новой версии SQL Server 2017 в Windows и Linux. Хотя общий рабочий процесс миграции для Windows и Linux является одинаковым, переход на SQL Server 2017 для Linux требуется несколько моментов.

### <a name="specifying-the-back-up-path"></a>Указание пути резервного копирования
Linux и Windows используют форматы другой путь. Таким образом переход на SQL Server 2017 в Linux требует, что пользователь предоставить путь к расположению файла версий Windows и Linux. Можно предоставить обе версии пути, по-разному в зависимости от расположения физического файла.
Если физический файл резервного копирования на компьютер под управлением:
- Linux, используйте «samba» предоставить совместный доступ совместное использование файла с другими компьютерами в сети.
- Windows, выполните команду «mnt», чтобы подключить общий ресурс на компьютер под управлением Linux.

> [!NOTE]
> Дополнительные сведения об использовании общей папки «samba» или команда «mnt» выходят за рамки этой статьи.

### <a name="migrating-windows-logins"></a>Миграция имен входа Windows
Во время миграции Active Directory (AD) учетные записи входа официально поддерживается SQL Server 2017 в Linux, его требуется дополнительная настройка для успешной работы. Ознакомьтесь со статьей [проверки подлинности Active Directory с SQL Server в Linux](https://docs.microsoft.com/sql/linux/sql-server-linux-active-directory-authentication) подробные сведения о настройке учетных записей Active Directory на SQL Server 2017 в Linux. После выполнения необходимой конфигурацией, установка будет завершена, и имена входа Active Directory можно перенести как обычно. Стандартная проверка подлинности SQL работает должным образом без дополнительных настроек.

## <a name="dma-v20"></a>DMA v2.0
В выпуске версии 2.0 DMA включает в себя улучшенные рекомендации функций базы данных Stretch для предоставления правильного приоритетных таблицы с максимальной экономии пространства хранения.

## <a name="dma-v10"></a>Версия 1.0 DMA
В выпуске версии 1.0 DMA является первоначального выпуска, а также предоставляет для:
- Обнаружение проблем, которые могут повлиять на обновление до локальной версии SQL Server. Описываются все результаты как проблемы совместимости, и они могут быть относится к категории в следующих областях:
    - Критические изменения
    - Изменения в поведении
    - Устаревшие функции
- Обнаружение новых возможностях целевой платформы SQL Server, базы данных могут использовать преимущества обновления. Описываются все результаты как рекомендуемые возможности, и они могут быть относится к категории в следующих областях:
    - Производительность
    - безопасность
    - Память
-   Современный пользовательский интерфейс для выполнения оценки.
